## Project Overview

A wasmCloud application (WebAssembly) that monitors network device uptime via HTTP health checks. It serves a REST API and live HTML dashboard. Written in Rust, compiled to `wasm32-wasip2`.

## Build & Run

```bash
# Build both component and service (compiles Rust → wasm)
wash build

# Development mode (builds, starts wasmCloud host, deploys automatically)
wash dev
```

The API listens on **http://localhost:8080** (configured in `.wash/config.yaml`).

### Prerequisites

- `wash` CLI v2.0+ (wasmCloud Shell)
- Rust with target: `rustup target add wasm32-wasip2`
- (Optional) Redis if using the `redis` KV backend — see `.wash/config.yaml`

## Architecture

Workspace with two crates that form a single wasmCloud workload:

### `component/` — HTTP Component
Exports `wasi:http/incoming-handler` (REST API + dashboard) and a custom `cron` interface. Uses three WASI capabilities:
- **`wasi:http/incoming-handler`** — serves the REST API (inbound HTTP)
- **`wasi:http/outgoing-handler`** — makes outgoing HTTP requests to poll devices
- **`wasi:keyvalue/store`** — persists device config, metrics, and check history
- **`cosmonic:uptime-monitor/cron`** — exported interface called by the service

### `service/` — Cron Service
A long-running service that imports the `cron` interface and calls:
- `poll-all` every **1 second** to check all configured devices
- `prune` every **60 seconds** to remove check history older than 24 hours

The WIT world is defined in `wit/world.wit` under package `cosmonic:uptime-monitor`. WIT dependencies are resolved automatically by `wash build` via `wkg`.

Capability provider configuration (HTTP server address, KV backend) lives in `.wash/config.yaml`.

## Key Patterns

- **Bindings**: Generated by `wit_bindgen::generate!()` in a `mod bindings` block — all WASI types come from there
- **KV keys**: Device list under `uptime:devices`, per-device metrics under `uptime:metrics:{device_id}`, check history under `uptime:history:{device_id}`
- **Routing**: Pattern matching on `(Method, path)` tuples in `UptimeMonitor::handle()`
- **Cron via WIT**: The service crate imports `cosmonic:uptime-monitor/cron` and loops with `std::thread::sleep`
- **No async runtime**: Component model is synchronous; outgoing HTTP uses `pollable.block()` for responses
- **No chrono**: Timestamps are hand-computed from `wasi:clocks/wall-clock` via `std::time::SystemTime`
